# Feature Specification: Per-page provenance with shadow git and structured checkpoint JSON

**Feature Branch**: `005-page-provenance`
**Created**: 2026-06-28
**Status**: Draft
**Input**: User description: "Per-page provenance. Today the InsightMesh pipeline produces rich data about which exchanges drove which wiki drafts, what Editor's signals and rationale were, which model produced which messages, and which agents contributed what to each checkpoint, but all of it lives only in transient session logs. Nothing on the wiki page itself records its source or the agents' reasoning. This feature persists provenance two ways: a structured per-checkpoint JSON file (the permanent system of record, queryable via jq, migratable to a database in Phase B) and a richer frontmatter block on each wiki page (at-a-glance summary plus a pointer to the JSON). A separate shadow git repo holds copies of page snapshots so users get git-log-style diff history of how each page evolved across edits. All bookkeeping is orchestrator-side; Editor's contract is unchanged. The session log stays as the operational/diagnostic artifact, separate from this permanent provenance record."

## Clarifications

### Session 2026-06-28

- Q: How are checkpoint JSON files named to avoid collisions across multiple conversations? → A: Per-conversation subdirectory under `.history/checkpoints/`. Each conversation gets its own subdirectory keyed by the conversation identifier (with the sentinel `_flat` for Spec 001 flat-array transcripts that have no conversation id, and filesystem-unsafe characters sanitized to hyphens per the Spec 004 cursor rule). Per-conversation `cp-NNN` numbering stays aligned with the Spec 004 cursor's `checkpoint_number`; no second numbering scheme is introduced.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Structured provenance per page and per checkpoint (Priority: P1)

When a checkpoint completes, the orchestrator writes a structured JSON file recording everything queryable about that checkpoint (conversation identifier, export path, provider, models used, transcript hash; per-exchange records with user and assistant message identifiers; per-page Editor decisions with full rationale, action, confidence, contributing exchange indices, and the signals dict; results summary; convenience links to the session log and cursor). Each wiki page the checkpoint touched gets a frontmatter `provenance:` block recording the latest checkpoint that touched it, the conversation identifiers that have ever contributed to it, the latest editor action and confidence, total edits, and cumulative exchange count.

**Why this priority**: This is the MVP that delivers all the observability and provenance value. With the structured JSON, users (and future tooling) can answer "where did this page come from", "which messages contributed", "what was Editor's confidence on this decision", and aggregate queries across many checkpoints. The git-diff layer (US2) is genuinely valuable but strictly enhancement; US1 alone is shippable.

**Independent Test**: After a checkpoint completes on a fixture transcript, assert that `<vault>/InsightMesh/.history/checkpoints/<conv-id-or-"_flat">/cp-001.json` exists, parses against the schema, contains the expected conversation block, per-exchange records with both message identifiers per exchange, and per-page editor decisions with rationale plus signals. Assert that each touched wiki page's frontmatter contains a `provenance:` block with the cumulative fields computed correctly (matching the just-completed checkpoint).

**Acceptance Scenarios**:

1. **Given** a fresh checkpoint run on a Claude export, **When** the pipeline succeeds, **Then** `.history/checkpoints/<conv-id>/cp-001.json` is written with `schema_version=1`, the conversation block populated (`provider="anthropic"`, `models_used=[]` for Claude exports), per-exchange records carrying `user_message_id` and `assistant_message_id` from the export, and `editor.decisions[]` including rationale, confidence, contributing exchange indices, and the full signals dict.
2. **Given** a checkpoint that creates 3 new wiki pages, **When** the pipeline succeeds, **Then** each of the 3 pages has a frontmatter `provenance:` block with `latest_checkpoint` pointing to the just-written JSON, `conversations` listing the single conversation identifier, `latest_action="created"`, `total_edits=1`, and `exchange_count` equal to the count of distinct exchange indices contributing to that page.
3. **Given** an existing wiki page with a prior `provenance:` block from an earlier checkpoint, **When** a new checkpoint updates that page, **Then** the frontmatter `provenance:` block updates cumulatively: `latest_checkpoint` advances to the new checkpoint, `conversations` becomes the union of prior and new, `total_edits` increments by 1, `latest_action` and `latest_confidence` reflect this checkpoint, and `exchange_count` grows to the size of the union of prior and newly-contributing exchange indices.
4. **Given** a checkpoint where Editor's structured output failed to parse but the pipeline still recovered (per Spec 004 recoverable-failure path), **When** the orchestrator writes the JSON, **Then** the JSON still records the conversation and exchange blocks correctly; `editor.decisions[]` MAY be empty or partial; the file remains well-formed and parses against the schema.
5. **Given** a multi-checkpoint conversation (Spec 004 long-chat handling), **When** both checkpoints complete, **Then** under `.history/checkpoints/<conv-id>/`, both `cp-001.json` and `cp-002.json` exist and a page touched by both has `total_edits=2` in its frontmatter.

---

### User Story 2 - Diff history via shadow git repository (Priority: P2)

The orchestrator initializes a separate git repository at `<vault>/InsightMesh/.history/` (distinct from any git the user runs on their vault) and commits each checkpoint atomically. Each commit includes snapshots of every page touched in that checkpoint plus the `checkpoints/<conv-id>/cp-<NNN>.json` written in US1. Users get `git log -p pages/<slug>.md` style diff history of how each wiki page evolved across edits, with structured commit messages naming the checkpoint identifier, conversation identifier, and pages touched.

**Why this priority**: A refinement on top of US1. US1 makes the data exist and queryable; US2 makes the page-content evolution browsable with familiar git tooling. Editor's existing update path preserves the "current" state of each page; US2 adds the "all past states" view that makes auditing and rollback meaningful.

**Independent Test**: Run two checkpoints back-to-back on the same conversation where the second updates a page from the first. Verify `<vault>/InsightMesh/.history/` is a git repository, contains two commits both greppable by their checkpoint identifier, and `git -C .history log -p pages/<a-page>.md` shows the actual diff between the two versions of the page.

**Acceptance Scenarios**:

1. **Given** a fresh checkpoint, **When** the pipeline succeeds, **Then** `.history/` is a git repository (initialized this run or pre-existing), and the checkpoint produced exactly one commit whose message subject contains `[InsightMesh checkpoint:cp-001 conversation:<id>]` and whose body includes a `Metadata: checkpoints/<conv-id>/cp-001.json` line and a list of pages touched (each annotated with action and confidence).
2. **Given** two checkpoints back-to-back on the same conversation, **When** both complete, **Then** `git log` shows two commits in order with monotonically increasing checkpoint identifiers, each greppable independently by its `checkpoint_id`.
3. **Given** a page updated across two checkpoints, **When** `git -C .history log -p pages/<slug>.md` is run, **Then** the output shows the actual diff between the first and second versions of the page.
4. **Given** `git` is not installed on the system, **When** the orchestrator reaches the commit step, **Then** the JSON and frontmatter from US1 are still written, a warning is logged to stderr naming the missing dependency, and the run still succeeds (exit 0).
5. **Given** `.history/` already exists as a git repository from a prior run, **When** the orchestrator runs again, **Then** init is a no-op and the new commit lands cleanly on top of the existing history.

---

### Edge Cases

- The wiki page being updated already has a `provenance:` block. The orchestrator merges cumulatively (union of conversations, sum of edits, union of exchange indices, latest fields from this checkpoint) rather than overwriting.
- The wiki page being updated does NOT have a `provenance:` block (legacy page from before Spec 005, or a user-authored page). The orchestrator treats the existing provenance state as empty and writes a fresh block.
- Existing page frontmatter is malformed YAML. The orchestrator logs a warning, leaves the page's frontmatter as-is, but still writes the checkpoint JSON so other pages and the structured store are unaffected.
- A page is touched by multiple conversations over its lifetime (cross-thread compounding). The `conversations` list in frontmatter grows accordingly; the JSON for each individual checkpoint still records only that checkpoint's single conversation.
- ChatGPT export with rich `models_used` versus Claude export with empty `models_used`. JSON faithfully records what the upstream parser surfaced; the orchestrator never fabricates model identifiers.
- Spec 001 flat-array transcript (no conversation identifier, no models, no per-message identifiers). The JSON records `provider=null`, `conversation.id=null`, `models_used=[]`, and per-exchange records with null message identifiers; the frontmatter `provenance:` block still computes cumulative fields correctly with `conversations=[]` (or omitted).
- The `.history/checkpoints/` directory exists from a partial prior write (e.g., crashed mid-write). New JSON writes use atomic write-temp then rename so a half-written file is never observed.
- Concurrent invocations on the same conversation. Same single-writer assumption as Spec 004's cursor; not supported by this spec; documented.
- A checkpoint produces zero drafts (Synthesis returned nothing). The JSON is still written with empty `editor.decisions[]` and empty `results`; no frontmatter updates happen because no pages were touched; no git commit happens because there is nothing to snapshot.
- A page slug would contain filesystem-unsafe characters on some platforms. The orchestrator sanitizes the snapshot filename using the same rules already used for vault page filenames.
- The shadow repo's `git commit` fails (disk full, permissions, hook error). The JSON and frontmatter from US1 have already landed; the commit failure is logged to stderr but the run still succeeds; the next successful checkpoint's commit will include both checkpoints' page changes because git handles untracked diffs cleanly.
- The user (or another tool) has modified `.history/` outside the orchestrator (manual commits, branch switches, dirty working tree). The orchestrator's commit step does not attempt to verify or repair the repository state; it stages and commits its own changes and emits whatever git emits. Documented as unsupported but non-destructive.
- A page Editor wrote in this checkpoint has been deleted (by the user or another tool) before the orchestrator's snapshot step runs. The orchestrator logs `[provenance] page disappeared before snapshot: <path>` to stderr, skips the snapshot for that page, but still records the page in the checkpoint JSON's `editor.decisions[]` (Editor's action is what happened; the page's later deletion is outside the pipeline's control). The frontmatter merge for that page is also skipped silently (the file does not exist to be opened). Run still exits 0.
- The source transcript's conversation identifier is literally the string `_flat`. Treated as a normal conversation identifier (no special-casing); the resulting subdirectory `<vault>/InsightMesh/.history/checkpoints/_flat/` then holds checkpoints both from this real-but-named-`_flat` conversation and from any flat-array transcripts encountered later. Collision is acceptable because (a) the probability of a real identifier matching the sentinel is vanishingly small for echomine-generated identifiers (UUIDs), and (b) per-checkpoint `cp-<NNN>` numbering remains correct within whichever conversation produced it. Documented for completeness; no special-case code is required.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST write a per-checkpoint JSON file at `<vault>/InsightMesh/.history/checkpoints/<conversation_subdir>/cp-<NNN>.json` after each successful checkpoint. `<NNN>` is the zero-padded (3 digits, e.g., `cp-002`) `checkpoint_number` from the Spec 004 cursor; the file's `checkpoint_id` field MUST equal `f"cp-{checkpoint_number:03d}"` (single derivation rule; the two fields cannot drift). `<conversation_subdir>` is the conversation identifier with filesystem-unsafe characters sanitized to hyphens (matching the Spec 004 cursor file naming rule), OR the sentinel `_flat` when the source transcript carries no conversation identifier (Spec 001 flat-array shape). The per-conversation subdirectory MUST be created if missing. The JSON write MUST be atomic against process interruption such that a reader observing the target path between writes sees either the prior fully-written file or the new fully-written file, never a partially-written file (write-temp-then-rename via `os.replace` or equivalent satisfies this). All `timestamp` and other datetime fields in the JSON MUST be timezone-aware UTC, serialized as ISO 8601 with the `Z` suffix (matching the Spec 004 cursor `updated_at` convention).
- **FR-001a**: Once written, a `cp-<NNN>.json` file is immutable. System MUST NOT overwrite, patch, or delete a `cp-<NNN>.json` file in normal operation. If the orchestrator attempts to write to a path where a `cp-<NNN>.json` already exists (e.g., re-running a checkpoint that was already written but whose cursor advance failed), the write MUST refuse (the atomic rename detects the collision) and the orchestrator MUST log to stderr per FR-016 without failing the run.
- **FR-001b**: All ordered collections inside a `cp-<NNN>.json` file (the `exchanges` array, the `editor.decisions` array, the `conversations` and `exchange_indices` arrays inside `ProvenanceFrontmatter`, the `pages_created` / `pages_updated` / `pages_skipped` arrays inside `results`) MUST be serialized in a deterministic order: `exchanges` and `editor.decisions` in the order processed; lists of strings (`conversations`, `pages_*`) sorted ascending; lists of integers (`exchange_indices`) sorted ascending. Deterministic ordering makes the JSON files diff-friendly across runs and lets `jq` queries assume stable iteration order.
- **FR-002**: The checkpoint JSON MUST carry an integer `schema_version` field. Within `schema_version=1`, future field additions MUST be additive (optional); readers MUST tolerate unknown extra fields and missing optional fields. A major version bump is reserved for incompatible changes and requires explicit migration tooling.
- **FR-003**: The checkpoint JSON's `conversation` block MUST record: `id` (string or null for flat-array transcripts), `export_path` (string), `provider` (`"anthropic"` | `"openai"` | null), `models_used` (list of strings; empty list when the upstream parser does not surface them), and `transcript_hash` (the same SHA-256 used by the Spec 004 cursor).
- **FR-004**: The checkpoint JSON's `exchanges` array MUST contain one record per exchange processed in this checkpoint, with `index` (integer, matching the exchange index used by Spec 004), `user_message_id` (string or null), and `assistant_message_id` (string or null). Null identifiers MUST be used when the source transcript does not carry them (Spec 001 flat-array shape).
- **FR-005**: The checkpoint JSON's `editor.decisions` array MUST contain one record per page Editor touched in this checkpoint, with `file` (page filename relative to the vault root), `action` (`"created"` | `"updated"` | `"skipped"`), `confidence` (`"high"` | `"medium"` | `"low"`), `rationale` (string), `exchange_indices` (list of integers), and `signals` (the full signals dict Editor used to make the decision). All values inside the `signals` dict MUST be JSON-serializable using the standard library `json` module (strings, numbers, booleans, null, lists of these, dicts with string keys of these). If a non-serializable value is encountered, System MUST coerce it via `repr()` and emit a stderr warning naming the offending key (better to record a degraded representation than to abort the provenance write).
- **FR-006**: The checkpoint JSON's `results` object MUST mirror the Spec 004 result categories: `pages_created`, `pages_updated`, `pages_skipped`, each a list of page filenames.
- **FR-007**: The checkpoint JSON's `links` object MUST carry convenience pointers to the session log and the cursor file (`session_log`, `cursor`), as repository-relative paths beneath the vault. Provenance reads MUST NOT depend on the linked targets existing; the pointers are advisory.
- **FR-008**: For every page Editor created or updated in this checkpoint, System MUST write or merge a `provenance:` block in the page's YAML frontmatter. The block contains: `latest_checkpoint` (a vault-relative POSIX path, forward slashes even on Windows, anchored at the vault root, pointing to the just-written `.history/checkpoints/<conversation_subdir>/cp-<NNN>.json` — e.g., `InsightMesh/.history/checkpoints/d126dc13-…/cp-002.json` — POSIX-only separators ensure the frontmatter stays portable when a vault is moved across platforms), `conversations` (list of conversation identifiers; empty list permitted when no identifier exists), `latest_action` (`"created"` | `"updated"`), `latest_confidence` (`"high"` | `"medium"` | `"low"`), `total_edits` (integer), and `exchange_count` (integer).
- **FR-009**: When the page already has a `provenance:` block from a prior checkpoint, System MUST merge cumulatively rather than overwrite: `conversations` becomes the union of prior and new; `total_edits` increments by 1; `exchange_count` becomes the size of the union of prior and newly-contributing exchange indices; `latest_checkpoint`, `latest_action`, and `latest_confidence` are taken from this checkpoint. `total_edits` counts the number of distinct checkpoints in which Editor took a `created` or `updated` action on this page; `skipped` actions never get a frontmatter merge and therefore never increment `total_edits`. For the `exchange_count` cumulative computation, System MUST read the prior `cp-<NNN>.json` pointed to by the page's prior `latest_checkpoint` to recover the prior contributing exchange indices, then union with this checkpoint's `exchange_indices`. If the prior `latest_checkpoint` pointer is missing, unparseable, or points at a non-existent file, System MUST fall back to `exchange_count = prior.exchange_count + len(this_indices)` (an upper bound that may overcount across re-runs; this fallback path MUST emit a stderr warning naming the page and the missing prior checkpoint).
- **FR-010**: When the page does not have a `provenance:` block, System MUST initialize one from this checkpoint with `total_edits=1`. When the page has frontmatter that fails to parse as YAML, System MUST leave the page's frontmatter as-is, log a warning to stderr naming the page, and still write the checkpoint JSON and (where applicable) the git commit.
- **FR-011**: All other existing frontmatter fields on the page (e.g., `title`, `created`, `updated`, `source`, `tags`) MUST be preserved unchanged when the orchestrator writes the `provenance:` block. The orchestrator MUST NOT reorder, drop, or rewrite any other frontmatter key. The frontmatter merge MUST be atomic at the page-file granularity: a reader (or a crash recovery) observing the page file between writes MUST see either the prior fully-written page or the new fully-written page, never a page whose frontmatter is partially merged. Write-temp-then-rename via `os.replace` in the page's parent directory satisfies this.
- **FR-012**: System MUST initialize a shadow git repository at `<vault>/InsightMesh/.history/` when one does not already exist. Three input states are possible: (a) the directory does not exist — System creates it and runs `git init`; (b) the directory exists and contains a `.git/` subdirectory — System treats it as already-initialized and proceeds without modifying configuration; (c) the directory exists but contains no `.git/` subdirectory (e.g., user manually deleted `.git/`) — System runs `git init` to re-initialize, preserving any existing files in the directory (git init is non-destructive). In all three cases, System MUST NOT reset, reconfigure, or otherwise modify an existing git repository's configuration.
- **FR-013**: After writing the checkpoint JSON and updating page frontmatters, System MUST copy each touched page to `<vault>/InsightMesh/.history/pages/<sanitized-slug>.md` and create one git commit in the shadow repository that stages the new/updated page snapshots plus the new `checkpoints/<conversation_subdir>/cp-<NNN>.json`. The shadow repository's working tree MUST stay confined to `.history/`; the user's vault pages and any user-managed git repository at the vault root MUST NOT be affected.
- **FR-014**: The shadow-repo commit message MUST follow a machine-greppable pattern: the subject line MUST contain `[InsightMesh checkpoint:<checkpoint_id> conversation:<conversation_id_or_"_flat">]` followed by a brief human-readable summary of pages touched; the body MUST include a `Metadata: checkpoints/<conversation_subdir>/cp-<NNN>.json` line and a per-page list with action and confidence. Bidirectional navigation between commit and JSON MUST work via the `checkpoint_id` string combined with the conversation identifier (no commit SHA is stored in the JSON).
- **FR-015**: When `git` is not available on the system, System MUST still attempt the checkpoint JSON write (FR-001) and frontmatter merges (FR-008), log a single warning to stderr naming the missing dependency, and exit `0`. When the JSON write or a frontmatter merge fails due to an OS error (disk full, permission denied, etc.), System MUST log the failure to stderr naming the affected path and continue; System MUST NOT raise the failure to the run level. In all provenance-only failure scenarios (no git, JSON write fails, frontmatter merge fails, shadow-repo commit fails), the run exit code is determined solely by the agent pipeline and the Spec 004 cursor write: a provenance failure NEVER changes the exit code.
- **FR-016**: When the shadow-repo commit step fails (git error, dirty working tree set up by the user, hook failure, permissions, disk error, etc.), System MUST log the failure to stderr but MUST NOT fail the run. The checkpoint JSON and frontmatter updates from FR-001/FR-008 have already landed; the next successful checkpoint's commit will sweep up the orphaned snapshot files together with its own.
- **FR-016a**: All provenance-related stderr messages MUST be single-line, prefixed with `[provenance] `, and machine-greppable. The required prefix lets users (and downstream tooling) filter for provenance signal without parsing the broader CLI output. Failure messages MUST name the affected artifact (page path, checkpoint id, conversation id, etc.) where applicable. The word "warning" in this spec means "stderr line with the `[provenance] ` prefix"; no log-level escalation, no structured logging library, no separate log file. Examples: `[provenance] git not on PATH; skipping shadow-repo commit`, `[provenance] frontmatter parse failed for InsightMesh/<page>.md: <yaml error>`, `[provenance] commit failed: <git stderr>`.
- **FR-017**: All provenance bookkeeping (JSON write, frontmatter merge, page snapshot, shadow-repo commit) MUST happen orchestrator-side, after Editor's pipeline returns successfully and before the Spec 004 cursor save. The Editor agent's prompt and structured-output contract MUST NOT change. Editor MUST NOT emit a `provenance:` block in its drafts; the orchestrator owns it. The four bookkeeping steps MUST execute in this order: (1) write the checkpoint JSON to its target path; (2) for each touched page, merge the `provenance:` frontmatter; (3) for each touched page, copy a snapshot to `.history/pages/<sanitized-slug>.md`; (4) initialize the shadow repo if missing and create the commit. Any step may fail without aborting subsequent steps for unaffected artifacts: a frontmatter merge failure on one page MUST NOT block the merge for other pages or the snapshot/commit steps; a snapshot failure on one page MUST NOT block other snapshots or the commit. Recovery contract on process interruption: any partially completed step leaves at most one page's frontmatter partially merged at the file-rename boundary (precluded by FR-011 atomicity), or one snapshot not yet committed (swept up by the next checkpoint's commit per existing edge-case behavior), or one JSON file not yet committed (same sweep behavior).
- **FR-018**: The session log MUST continue to be written per Spec 001/004 behavior. The session log and the checkpoint JSON are distinct artifacts with deliberate overlap on Editor decisions and page results; neither replaces the other.
- **FR-019**: Provenance bookkeeping MUST NOT extend the wall-clock runtime of a checkpoint by more than a small constant overhead relative to the agent work (target: under 1 second per checkpoint of provenance-only work on a vault with hundreds of pages). Provenance failures MUST NOT block agent work or cursor advancement; the cursor write remains the terminal step of a successful checkpoint.
- **FR-020**: System MUST NOT support provenance reads that require a checkpoint JSON write to have linked the commit SHA. The JSON is self-sufficient for provenance queries; navigation to the corresponding commit is via grep on `checkpoint_id`.
- **FR-021**: System MUST be resilient to process kill (SIGTERM, SIGINT, hard crash) at any point during the provenance bookkeeping sequence. The FR-001 atomic write, FR-011 frontmatter atomicity, and existing edge-case behavior (next commit sweeps up orphaned snapshots) collectively mean a killed run leaves the on-disk state consistent at the file granularity. On the next invocation, no recovery action is required: the killed checkpoint's cursor was not advanced (Spec 004 cursor write happens after all provenance steps per FR-017), so the next run re-processes the same exchanges and re-writes the same `cp-<NNN>.json` (refused by FR-001a if already fully written, otherwise overwriting the killed temp file). System MUST NOT perform special-case recovery scans on startup.
- **FR-022**: System MAY detect orphaned shadow-repo state on startup (uncommitted page snapshots in `.history/pages/`, or `.history/checkpoints/*/cp-*.json` files newer than the latest shadow-repo commit) and emit a single informational `[provenance] ` stderr line summarizing what was found. System MUST NOT block, refuse, or alter behavior based on the detection; the next successful commit sweeps up orphans by design (FR-016). Detection is optional and best-effort; absence of the detection logic does not violate this requirement.

### Key Entities

- **Checkpoint record**: One JSON file per successful checkpoint at `.history/checkpoints/<conversation_subdir>/cp-<NNN>.json` (with `<conversation_subdir>` being the conversation identifier or the `_flat` sentinel for transcripts that lack one). Permanent, append-only system of record for provenance. Self-sufficient: provenance queries MUST NOT need to traverse the `links.*` pointers to be answered. Contains the conversation block, the exchanges array, the editor decisions array, the results summary, and the convenience links. Together with the `Page provenance block`, this constitutes the read contract consumed by external viewers (planned: dedicated in-Obsidian viewer plugin in a separate repo).
- **Page provenance block**: A `provenance:` key inside an existing wiki page's YAML frontmatter. At-a-glance summary keyed for Obsidian Dataview / Bases queries without parsing JSON files. Carries `latest_checkpoint`, `conversations`, `latest_action`, `latest_confidence`, `total_edits`, `exchange_count`. Updated cumulatively across checkpoints. The `latest_checkpoint` pointer is the anchor a viewer follows from the open wiki page to its full provenance record.
- **Shadow history repository**: A git repository at `<vault>/InsightMesh/.history/`, distinct from any git the user runs on the vault root. Holds the JSON files and page snapshots. One commit per successful checkpoint. Purely for diff history and audit; the orchestrator never reads back from prior commits.
- **Page snapshot**: A copy of a touched wiki page written to `.history/pages/<sanitized-slug>.md` as part of the same shadow-repo commit. Lets `git log -p pages/<slug>.md` show the page's evolution.
- **Editor decision (provenance form)**: A record inside `editor.decisions[]` of a checkpoint JSON describing one page Editor touched: which file, what action, what confidence, what rationale, which exchanges contributed, and the full signals dict used. Mirrors information from the session log but is the permanent record.
- **Conversation block (provenance form)**: The `conversation` object inside the checkpoint JSON: source export path, conversation identifier, provider, models used, transcript hash. Always recorded for the single conversation that drove this checkpoint, even when a page accumulates across multiple conversations.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After any successful checkpoint, the user can open `<vault>/InsightMesh/.history/checkpoints/<conversation_subdir>/cp-<NNN>.json` and answer the following questions without consulting the session log: which conversation produced this checkpoint, which exchanges were processed, which pages were created/updated/skipped, what Editor's rationale and confidence were for each decision, and which signals drove each decision.
- **SC-002**: After any successful checkpoint, every page Editor created or updated carries a `provenance:` frontmatter block whose `latest_checkpoint` resolves to the just-written JSON file (under the correct per-conversation subdirectory) and whose `total_edits` equals the count of distinct prior-checkpoints-plus-this-one in which Editor touched the page.
- **SC-003**: For a conversation processed across two checkpoints where the second updates a page from the first, `git -C .history log` shows exactly two commits attributable to this conversation (one per checkpoint) and `git -C .history log -p pages/<slug>.md` shows the actual diff between the two versions of the page.
- **SC-004**: Running the batch command on a vault that already has a populated `.history/` (from prior checkpoints) advances cleanly: no init errors, no overwrite of prior JSON files, monotonically increasing checkpoint numbers, monotonically advancing commit history.
- **SC-005**: When `git` is uninstalled (or fails to commit for any reason), the run still exits 0; the JSON file and frontmatter updates are still on disk; only the shadow-repo commit is missing. The user can rerun on the next checkpoint or install git and continue without data loss.
- **SC-006**: Provenance bookkeeping adds under 1 second of wall-clock time per checkpoint on a vault with hundreds of pages (measured separately from agent work). No regression to checkpoint correctness or cursor advancement.
- **SC-007**: A reader written against `schema_version=1` continues to parse later checkpoint JSONs that have added additional optional fields, without error. The schema is forward-compatible by construction.

## Assumptions

- The vault is on a local filesystem the orchestrator can write to. Remote / cloud-only vaults (e.g., live Notion bridge) are out of scope for the shadow-repo write path.
- The user is not running InsightMesh concurrently on the same conversation. The Spec 004 single-writer assumption extends to the provenance writes; concurrent invocations on the same conversation are not supported.
- The user is the only writer to `.history/`. Manual user commits or branch switches inside `.history/` are not supported by this spec but are non-destructive to the orchestrator's own writes.
- `echomine` (the export parser) surfaces per-message identifiers and per-conversation models-used metadata in the version pinned by this project (1.5.0+). For older transcripts or the Spec 001 flat-array format, the corresponding fields are null/empty per FR-003 and FR-004.
- The Editor agent's existing structured-output contract (Spec 001 `EditorDecision`) already carries the action, confidence, rationale, exchange indices, and signals dict that the checkpoint JSON records. The orchestrator captures these in memory during the existing Editor invocation and does not require Editor prompt changes. Any change to `EditorDecision`'s action / confidence / rationale / exchange_indices / signals shape MUST be treated as a breaking change for this spec and trigger a coordinated update to `src/history.py` model definitions, test fixtures, and (if the JSON-visible shape changes) the checkpoint JSON `schema_version`.
- The user's vault is small enough (single-digit thousands of pages) that page-snapshot copies into `.history/pages/` per checkpoint do not meaningfully tax disk space. A Phase B housekeeping mode for shallow / pruned shadow history is deferred.
- A failed shadow-repo commit (FR-016) leaves no half-staged state that would block a future commit. Git's own atomicity guarantees plus the orchestrator's `git add` of explicitly-named files (no `git add .` on user-modified state) are sufficient.

## Out of Scope

- **In-Obsidian viewer plugin**. A read-only Obsidian plugin that renders this spec's on-disk artifacts (the `cp-<NNN>.json` files and the frontmatter `provenance:` blocks) into a side-pane view with click-through navigation and diff rendering between page snapshots. Lives in a **separate repo** ([aucontraire/insightmesh-obsidian](https://github.com/aucontraire/insightmesh-obsidian)). Obsidian plugins are TypeScript + esbuild and have their own distribution channel (BRAT for beta, the Obsidian community plugin registry for general availability). The on-disk contract from this spec (FR-001 through FR-014, the additive schema rule from FR-002) IS the plugin-facing read contract; no spec changes are required to enable the plugin. Tracked as the next planned spec; not part of Spec 005.
- **Obsidian Dataview / Bases templates** for in-vault display of provenance fields. The spec ships the data shape; example templates may ship as docs but no user installation is required for this spec to deliver value.
- **Intra-message attribution** (which sentence in a single message landed in which paragraph of the wiki). Requires an inline-citation agent; Critic-agent territory; separate spec. Documented as a known limitation.
- **Contradiction detection** (Editor flags when new content disagrees with existing wiki pages on shared topics). Different spec; can read provenance as input but does not depend on this work to ship.
- **Phase B Postgres migration** of the checkpoint JSON. The JSON schema is the migration source; the migration tool itself is Phase B work.
- **Multiple editor decisions for the same page within a single checkpoint**. Editor's current contract emits one decision per draft per checkpoint; this spec does not loosen that.
- **Removing or deprecating the session log**. Rejected design alternative. The session log remains the operational / diagnostic artifact; the checkpoint JSON is the permanent provenance record. Both coexist with deliberate overlap on Editor decisions and page results.
- **Embedding the git commit SHA in the JSON** via a two-commit or hash-prediction pattern. Rejected design alternative; the `checkpoint_id` is the bidirectional join key and navigation works in both directions via grep without write-ordering contortions.
- **Strict-refuse schema versioning** (the Spec 004 cursor pattern, where any unknown `schema_version` aborts the read). Rejected for permanent records: the checkpoint JSONs are non-regeneratable history. Schema evolves additively within v1; readers tolerate unknown-extra and missing-optional fields. Major version bumps are reserved for incompatible changes and require an explicit migration script.
- **A separate "history" Pydantic model layer for in-memory representation beyond what's needed to serialize/parse the JSON**. Keep it minimal: the JSON is the wire format; the orchestrator uses transient Pydantic models where they pay for themselves (strict input/output validation) and plain dicts where they do not.
- **Pruning, compaction, or garbage-collection of `.history/`** (e.g., shallow clones, squash-merge older checkpoints, drop snapshots older than N). Phase B housekeeping; not required for Phase A correctness.
- **Multi-process / multi-user safety** on the shadow repository (file locks, transactional writes). Same posture as the Spec 004 cursor: single-writer assumption documented; remediation is a future concern.
- **A separate Python read library or CLI command for provenance** (e.g., `insightmesh history <page>` as a CLI subcommand, or a `mesh.history` Python module for programmatic reads from other Python tools). The in-Obsidian viewer plugin (above) addresses the human-inspection use case; a typed Python read surface for scripts and pipelines is a separate concern, deferred to a future spec. Until either ships, scripting workflows use `jq` on the JSON files, shell `git -C .history log` on the shadow repo, and YAML readers on the frontmatter.
- **Major-version-bump migration tooling**. FR-002 reserves the right to bump `schema_version` for incompatible changes but explicitly defers the tooling to whatever future spec requires the bump. Until that point, all schema evolution stays within `schema_version=1` via additive optional fields, and no migration tool exists. Major bumps should be exceptional, not routine: the cost of providing migration tooling is the gate that disciplines us toward additive evolution.
